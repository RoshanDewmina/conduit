You are the senior engineer taking Lancer's **Governed Approvals v1** through a full pre-App-Store-submission
pass: examine the codebase in depth, fix what's broken, verify the whole thing actually works on the simulator,
assess submission readiness, and make the app measurably simpler and more intuitive. Work **autonomously** ŌĆö
make every change you judge correct, verify it by building and testing, and report at the end. This is a large,
long-horizon task; plan it up front, use high effort, and use subagents and your session memory file liberally.

## North star (the primary objective)
**The app must stay straightforward and intuitive.** Every UX/UI decision is judged against that. When you
propose or make a design change, the test is "does this make the core flow easier to understand and act on?" ŌĆö
not "is this more featureful or more decorated."

## Operating contract / guardrails (do not violate)
- **Scope = IN:** the iOS app (`Packages/LancerKit/Sources/**` + the Xcode targets: `Lancer`,
  `LancerWidget`, `LancerLiveActivityWidget`, `LancerWatch`, `LancerWatchWidget`), the approval-relay
  **Go** backend at `daemon/push-backend/**`, and the planning/readiness docs under `docs/**` and `ship-plan/**`.
- **Scope = OUT (do not modify):** `marketing/**`, `docs/marketing/**`, `docs/website/**`,
  `docs/lancer-ui-prototype/**`, any `.claude/worktrees/**`, and the `daemon/lancerd` / `daemon/agent-runner`
  resident-daemon code unless a bug there directly breaks the governed-approvals flow (if so, flag it, don't
  silently rewrite it).
- **The Xcode project is generated by XcodeGen from `project.yml`.** Change targets / build settings / entitlements
  wiring in `project.yml` (and regenerate), **never hand-edit `Lancer.xcodeproj`**.
- **Autonomy:** for minor choices (naming, copy, spacing, which of two equivalent fixes), pick a reasonable
  option and note it ŌĆö don't stop to ask. **Ask first only for genuinely irreversible / outward-facing actions:**
  `git commit` or `git push`, opening a PR, any App Store upload/submission, rotating or printing secrets, deleting
  files you didn't create, or destructive backend/cloud operations. Do **not** submit to the App Store ŌĆö produce a
  go/no-go instead.
- **Security:** never print secret values; secrets live in Keychain / `~/.hermes/.env`. Keep the **TOFU host-key
  prompt intact in production paths** ŌĆö the debug gallery/live-SSH harness auto-trusts the first host key, and that
  behavior must never leak into a non-debug code path. Don't weaken ATS, entitlements, or auth to make something
  "work."
- **Tooling:** prefer the configured MCP servers over raw shell (see below). For any JS/TS tooling use `bun`,
  never npm/yarn/pnpm. For the Go backend use `go build` / `go vet` / `go test ./...`.
- You may install developer dependencies needed to build and test (simulators, `xcodegen`, `fastlane`, the Go
  toolchain) ŌĆö prefer project-local installs, and **list everything you installed** in the final report.

## Phase 2 ŌĆö Feature inventory & frontend-coverage audit
Study the codebase and build a complete **feature/capability inventory**, then verify each capability is actually
**represented and reachable in the UI**. This catches the "we built it but never surfaced it" gap (and the reverse).
1. Enumerate capabilities from source, not from docs: every LancerKit feature module (`AgentKit`, `InboxFeature`,
   `SessionFeature`, `SettingsFeature`, `KeysFeature`, `FilesFeature`, `DiffFeature`, `WorkspacesFeature`,
   `OnboardingFeature`, `NotificationsKit`, `SyncKit`, `SecurityKit`, ŌĆ”), every public view / route / navigation
   destination, every Settings toggle and feature flag, every backend endpoint/capability in `daemon/push-backend`
   (agents, decisions, dispatch, billing, credits, entitlements, artifacts), and the widget / watch / Live Activity
   surfaces.
2. Build a **coverage matrix** in `docs/audit/FEATURE_COVERAGE.md`: capability ŌåÆ where it's surfaced in the UI
   (which screen, and how a user reaches it) ŌåÆ reachable in the normal flow? (yes / only via debug gallery / not at all).
3. Flag drift in **both directions** and recommend a fix for each: **(a) implemented-but-hidden** ŌĆö a code or
   backend capability with no UI entry point a real user can reach; **(b) shown-but-unimplemented** ŌĆö UI that
   references a feature which is stubbed, dead-ends, or doesn't work. Implement the clear low-risk fixes (e.g. add
   a missing nav entry, remove a dangling button); flag the rest with a recommendation.

## Phase 3 ŌĆö Exhaustive static review, debug & hardening
Review **every in-scope module** for real bugs ŌĆö do not sample. Cover logic errors, edge cases, race conditions
and actor-isolation issues (Swift 6 strict concurrency), force-unwraps / out-of-range indexing, error handling and
**silent failures**, retain cycles, and security issues ŌĆö across all in-scope LancerKit modules and the Go relay
backend. Fan out with one subagent per module group plus one for the backend so coverage is complete, then run a
second **adversarial pass**: for each candidate bug try to *disprove* it (is it actually reachable? already
guarded?) before reporting, and keep hunting until a full pass turns up nothing new.
**Highest priority ŌĆö the governed-approvals path:** the approval card UI, decision capture, the live-SSH decision
channel, and the **backend-relay fallback** when there's no live channel (`daemon/push-backend/decisions.go`,
`approval_push_test.go`, `dispatch*.go`, `entitlements.go`). Confirm a decision made in the app reaches its target
**exactly once**, can't be spoofed or replayed, and fails safe.
Fix what you find; add or extend tests (`go test`, LancerKit tests) to lock in each fix. Log every finding with a
severity and a one-line reachability note in `docs/audit/FABLE_FINDINGS.md`.

## Phase 4 ŌĆö Full end-to-end run, touching every screen
Using the Phase-2 coverage matrix as the checklist, do a **clean-install run that exercises the whole app from
start to finish and visits every screen / route** ŌĆö not just the happy path. Screenshot each screen (save under
`docs/audit/screens/`, named by route) and assert on-screen state via the `ios-simulator` accessibility tree, not
by eyeballing pixels.
- **Primary flow:** first launch ŌåÆ onboarding ŌåÆ add/import key (KeysFeature) ŌåÆ connect to a host (the TOFU prompt
  must appear) ŌåÆ governance IA: **Inbox ŌåÆ Fleet ŌåÆ Activity ŌåÆ Settings** ŌåÆ receive and act on an **approval**
  (exercise **both** the live-SSH decision path and the backend-relay fallback) ŌåÆ drill into a **terminal session**
  at depth ŌåÆ background / foreground (Live Activity / widget if wired).
- **Then sweep everything else:** every Settings sub-screen and toggle, key management, files / diff / preview,
  workspaces, notifications, and every modal / sheet / detail you can reach ŌĆö plus each screen's **empty, loading,
  and error states**. Tick each screen off the coverage matrix; anything you can't reach in the normal flow is a
  Phase-2 gap to flag.
- Capture **light and dark** appearances (`xcrun simctl ui booted appearance dark|light`).
- Note every dead end, confusing label, missing empty/error/loading state, and any step that needs explanation.

## Phase 5 ŌĆö UX / UI / performance polish (against the north star)
1. **Assess, then propose before restyling.** List the friction points from Phase 4, then for each propose a
   concrete simplification with a one-line rationale tied to "simpler & more intuitive." Follow the **existing
   design system** (`Sources/DesignSystem/Tokens.swift`, `Components/`, `DebugGalleryView.swift` as the canonical
   reference) ŌĆö refine within it; do not invent a new visual language or introduce generic AI-template aesthetics.
2. **Implement the high-confidence improvements** (clear wins: copy, hierarchy, empty/error/loading states,
   tap-target size, the approval card per WS-11, nav/safe-area per WS-13). Re-screenshot **before/after** for each.
3. **Theme consistency:** audit every in-scope screen for token usage ŌĆö no hardcoded colors/spacing/fonts that
   bypass `Tokens.swift`, consistent light/dark, fixed-geometry list rows (see the badge-slot invariant in
   CLAUDE.md). Fix drift.
4. **Accessibility (WS-9):** Dynamic Type, VoiceOver labels/traits, contrast, hit targets. Fix the cheap, clear ones.
5. **Performance:** cold-launch time, scroll smoothness on the main lists, animation jank (PixelBox etc.),
   main-thread blocking, retain cycles in the SSH/session pipeline. Fix what's clearly wrong; flag the rest.

## Definition of done

  `daemon/push-backend` tests pass.
- A **feature coverage matrix** maps every code/backend capability to its UI surface; every implemented-but-hidden
  and shown-but-unimplemented gap is either fixed or flagged with a recommendation.
- **Every screen / route** has been visited and screenshotted in light and dark (per the coverage matrix), and the
  governed-approvals decision is verified end-to-end on **both** the live-SSH and relay-fallback paths.
- The exhaustive bug pass has **converged** ŌĆö a full review pass turned up nothing new ŌĆö and the governed-approvals
  path has no known correctness or security blockers (or they're flagged with a clear reason and proposed fix).
- A submission go/no-go exists with a concrete blocker list and the owner-only remainder.
- `docs/audit/FABLE_REPORT.md` + `docs/audit/FABLE_FINDINGS.md` + `docs/audit/FEATURE_COVERAGE.md` +
  `docs/audit/screens/` are written; the app is visibly simpler/clearer than the baseline, within the existing
  design system.
