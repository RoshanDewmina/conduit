## Simurgh feedback (2 items — owner priority, dogfooded co-deliverable)

1. **Lease reclaimed mid-use while a real, still-running process (`xcodebuild test-without-building`) held it, with no warning or error surfaced to the caller.** `lease-187` was acquired at 10:26:39 with a 30-minute TTL and `heartbeatAt` auto-advanced to 10:41:37 in the daemon's own bookkeeping (visible via `simurgh list --json`), yet by ~10:42 the underlying simulator device (`2A91DC88-2B1E-48D8-A506-AC7584BCC0F6`) had been torn down entirely (vanished from `xcrun simctl list devices`) and the lease state flipped to `"released"` — while my `xcodebuild test-without-building` was actively mid-run against it (it died with `Test crashed with signal kill before establishing connection` / `DTXMessage ... Failed to send signal 19`). Root cause suspected: I invoked `xcodebuild` directly via Bash rather than through `simurgh exec <lease-id> -- <cmd>`, so the daemon had no signal that the lease was in active use and reclaimed it under build-slot pressure (`buildSlots: 2`, 5 concurrent sweep lanes each running Xcode). **This should not be possible without at least a `renew`-vs-real-death distinction, or `exec` should be effectively mandatory (a hard requirement in AGENTS.md, not just a suggestion) for anything longer than a few seconds.**
2. **`simurgh exec` is incompatible with a caller-supplied `-derivedDataPath`.** `simurgh exec lease-188 -- xcodebuild test-without-building ... -derivedDataPath /tmp/dd-untested-sweep-e-tests ...` failed immediately with `xcodebuild: error: option '-derivedDataPath' may only be provided once` — `exec` unconditionally injects its own lease-scoped `-derivedDataPath` (per the tool's own `env` output) and duplicates the flag. This makes `exec` unusable for exactly the case where you need to run a **prebuilt** test bundle located in a specific, chosen derived-data path (e.g. re-running the same `.xcresult`-producing build across pairing-code retries) — the documented safe pattern (route long commands through `exec`) and the documented feature (control your own `-derivedDataPath`) are mutually exclusive today. Worked around by running xcodebuild directly and manually looping `simurgh renew` every 8s for the run's duration — which worked, but shouldn't be the load-bearing path.

## Top surprises

- **HID taps are dead exactly as warned** — a byte-identical `ui_describe_all` before/after a tap on a fully valid, correctly-addressed button (`Continue`, frame center (201,784)) confirmed this in one shot, no wasted retries needed once switched to XCUITest.
- **The terminal-open failure isn't a crash or a hang — it's silent.** The app just... stays on Workspaces. No error toast, no "couldn't connect" sheet, nothing in the UI distinguishes "still connecting" from "gave up." The only visible signal is the passive "Machine unreachable" label buried under Agents, which a real user opening Trusted Machines → Open Terminal would very plausibly never see before giving up.
- **Phone-side "paired" and daemon-side "paired" are not the same fact**, and nothing in this build cross-checks them before optimistically adding a machine record. That gap looks like the direct cause of the terminal-open FAIL, and is a strict superset of the previously-known "key mismatch" / "empty-code 400" pairing bugs logged in project memory (`project_relay_empty_code_400_root_cause`, `project_relay_decision_return_path_fixed`) — this may be the same underlying fragility resurfacing on a fresh code path (auto-pair) that those earlier fixes didn't cover.
- Simurgh's own lease bookkeeping (`heartbeatAt` auto-advancing) gave false confidence that lease-187 was fine right up until the device was gone — the JSON state didn't flag impending reclamation at all.
