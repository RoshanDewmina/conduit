How someone actually uses it

Getting a second opinion, then living with what it finds

A

Ask a different vendor to check the work

From a finished result, Verify with… opens a plain vendor sheet — Codex, OpenCode, Kimi — each row noting why it's a reasonable pick (resume+diffs access, local-first, or "verified 4d ago" recency). No modal takeover; this sits inline under the result, the same contextual placement GitHub uses for "Request reviews."

B

It agrees — stay quiet

The reviewer's card reads "no issues found" plus exactly what it checked (ran the tests, read all 5 files) — a corroboration card, not a badge. Accept is primary; nothing else demands attention.

C

It disagrees — get loud, get structured

A dedicated review card replaces the quiet corroboration layout: Confirmed (what still holds), Flagged (the actual concern, with a file:line), Also checked (scope of the review) — then the concrete diff underneath. Primary action routes back to the original author, not to the reviewer.

D

Compare two attempts from the same vendor

Two panes, same metrics shape (lines changed, approach, proof status) — a one-line recommendation, then Keep A / Keep B. This is peer-attempt comparison, not cross-vendor review.

E

Turn a passing proof into a tripwire

Right after a proof passes, one offer: save this as a future regression. Accepting adds it to a watched-flows list that reruns automatically next time this code path is touched.

F

Scrub back, fork, or export

A timeline of checkpoints (each with a worktree snapshot) plus attached Clips recordings. Fork starts a new mission from that timestamp; Export proof hands out a portable lancer.proof file instead.

G

Same mission, other surfaces

Watch shows an Emergency Stop next to active-agent count; a widget surfaces the single highest-priority "needs you"; Lancer Mac hands off to the exact hunk and scroll position; Siri routes a spoken instruction to a specific card, not a generic launch.

H

Pick the right vendor for the repo

Vendor Performance comparison ranks agents by proof-pass rate and revert-rate for the same repo/task type — side-by-side panes, then per-task-type drill-downs. Answers “who should I launch on for Swift UI vs daemon work?” without guessing.

I

Unbroken audit chain across vendor switches

Continuous Cross-Vendor Audit is an optional always-on mode: when you hand off from Claude to Codex mid-mission, the hash chain continues with no gap. Distinct from one-tap Verify with… — this is for compliance buyers who need provable continuity.

What Mobbin taught us, and why

Four references shaped the verify redesign, three more shaped D/F/G

GitHub inline "Request reviews"Contextual, not modal

The reviewer request lives inline under the PR, not in a takeover modal. Directly why Verify with… is a bottom sheet over the existing result card rather than a new full screen.

Gemini corroboration cardQuiet agree state

A second opinion that agrees should read as calm confirmation, not a celebratory badge. This is the direct model for panel B's understated "no issues found" card.

Sift "Incorrect." interstitialLoud disagree state

A full-attention, structured interstitial for when an automated checker actually disagrees. Source for panel C's dedicated review card and its Confirmed/Flagged/Also-checked structure, instead of quietly annotating the same layout as B.

Fabric model pickerVendor sheet

A plain list of model/provider options with a one-line reason per row. Panel A's three vendor rows (resume+diffs, local-first, recency) follow this shape rather than inventing a comparison table.

Binance "Compare Top Lead Traders"Metrics side-by-side

Two columns of the same metric set with the stronger number visually distinguished. Confirms panel D's shape: same fields per pane (lines, approach, proof), not a free-form diff between two unrelated attempts.

HBO Max "Key Moments" scrubberLabeled checkpoints

A vertical scrub rail with named, tappable markers instead of a bare progress bar. Maps directly onto panel F's timeline: checkpoints are named events (worktree snapshot, clip attached), not just timestamps.

Apple Fitness workout controlsWatch stop action

A single unambiguous stop control sized for a glance-and-tap on a small screen. Basis for panel G's Watch mini showing Emergency Stop as the one action that matters there.

Why this shape, concisely

Sequential and asymmetric, on purpose

Cross-vendor verify is deliberately sequential and asymmetric: one vendor produces a result, a different vendor only critiques it — it never re-solves the task. That's a distinct job from Run Comparison (panel D), where the same vendor produces two peer attempts and a human picks between them, and from Multi-Agent Showdown — parallel vendors racing the same task — which stays a named-but-undrawn gap, and only resurfaces as its own screen if cross-vendor verify ships first.

The asymmetry is the whole point: "agents double-checking each other's work, not the same class of model that did the work." A same-vendor re-check can share blind spots with the original attempt; a different vendor's CLI, trained differently, is structurally more likely to catch what the first one missed. Codex-mobile and Claude-mobile are single-vendor by construction and cannot offer this at all — it is a differentiator only a multi-vendor control plane like Lancer can build.

iOS 27 / WWDC 2026 — what applies here

This section already has a platform panel — three finds sharpen it further

PF

"Phone-a-friend" as a lighter verify path

LanguageModelSession.DynamicProfile · WWDC26 Session 242

Apple's own Dynamic Profiles API lets a tool spawn a short-lived, separate-transcript sub-session and return its answer as a string — literally named "phone-a-friend." A plausible, low-effort shape for panel A/B/C when the check doesn't need a full second CLI dispatch: an on-device model critiques the diff instead of invoking another vendor's agent. Worth prototyping as a lightweight tier under the real cross-vendor path, not a replacement for it.

XL

The widget panel's actual widget family

.systemExtraLargePortrait · WWDC26 Session 277

The prior research's "full-screen widgets" claim doesn't hold up — the real, confirmed addition is this new large-portrait family (from visionOS 26, now iOS/iPadOS/macOS 27), same data source as a smaller widget, bigger presentation. Directly the right target for panel G's widget mini showing "2 need you · proof ready" as a richer Home Screen tile, not an imagined lock-screen-covering surface.

DI

A fuller strip when propped up on a desk

isDynamicIslandLimitedInWidth · WWDC26 Session 223

New environment value that tells a Live Activity whether the Dynamic Island is width-constrained in landscape. A clean signal for a "phone propped up on a desk while away" mission strip: show the fuller phase/risk/elapsed layout when unconstrained, collapse to icon-only otherwise — better than guessing from orientation alone.
