How someone actually uses it

One workspace, top to bottom

A

Open the app — something needs you

You land on Home for whichever workspace you were last in (conduit, in this case) — scoped to one repo, not a global inbox. Needs you sits at the very top: one row, "Approval waiting · High risk · patch · hermes-box," with a single "Review" tap target. Below it, Today shows finished and running work as quiet rows — a checkmark, a diffstat, nothing louder.

B

Open the app — nothing needs you

This is the state you'll see most once the app is working well, so it doesn't go blank. "No decisions right now" stays as a row, "Start work from here" nudges toward the composer, and Yesterday's activity stays visible underneath. Calm, but never empty.

C

Start something

Tap the composer ("Plan, ask, build…") and it expands into a sheet with two inline selectors sitting above the text field — which repo/branch, and where it runs. Type once; everything else is two taps.

D

Pick where it runs

Tapping the run-target selector opens "Run on": your active target first (Cursor Cloud, in this example), then more machines and Remote Control/SSH below. Recent and relevant before exhaustive.

E

Pick which agent

Same shallow pattern for the model picker — a search field, the active model checked, more options below. No modal-within-modal, no separate router screen.

What Mobbin taught us, and why

Every reference maps to one decision

GitHub InboxRow shape

Actionable feed — repo, age, title, one status snippet, swipe-to-done. This is why Needs you is one plain row with a single action, not a card with a headline number.

GitHub "All done here!" & Todoist Today-emptyAll-clear state

Calm empty states that stay structured instead of going blank or cartoonish. This is why Panel B keeps showing rows — "No decisions right now" is still a row, not a void.

Linear InboxStatus dots

Dense triage with small unread dots instead of heavy badges. The colored .dot convention (active / warn / danger) used on every row across the entire board started here.

incident.ioRisk without panic

Severity, owner, and time shown cleanly, with no outage-siren styling. Why "high risk" reads as a text label plus a small warn dot — never a red banner. Risk should read as information, not alarm.

Slack workspace flow & Notion sidebarContext placement

Workspace context kept separate from the task list itself. Why "which repo/machine am I in" lives in the page title and composer selectors, never repeated on every row.

Why this shape, not the old one

The ledger fixes a real bug, not just a look

The previous Home was a dashboard: a greeting, a big "4 agents need you" headline, a machine-card stack up front. That's not only noisier — it was hiding a real defect the ledger model closes by construction.

!
Split-brain attention · documented, not hypothetical

The old headline read from pendingApprovalCount while the attention cards read from a separate FleetStore.attentionItems projection — two sources that can disagree, so the app could shout "4 agents need you" while showing zero actionable rows underneath. The ledger's headline, rows, and sidebar badge are specified to share one attention projection, so they can't desync.

The composer-with-inline-selectors model is the same instinct applied to starting work: no routing wizard, no separate "choose your agent" screen — one input, two taps to constrain it, matching how Cursor treats the same moment.

iOS 27 / WWDC 2026 — what applies here

Three concrete opportunities, none of them gating V1

Cross-checked against the full platform research pass. Most iOS 27 findings apply to Proof Suite or Work Thread, not Home — these three are the ones that actually touch this screen.

VA

Siri can reference a specific row, not just the app

.appEntityIdentifier(EntityIdentifier(for:identifier:)) · View Annotations API

Apple's own guidance is to use this exact modifier when multiple meaningful items are visible at once — precisely what Needs you / Today / Yesterday are. Tag each row and "Hey Siri, review the high-risk one" or "approve the checkout patch" could resolve to that specific row instead of just opening the app. No third-party app has shipped this yet (iOS 27 is still in beta), so treat as prototype-worthy, not proven.

AA

The same risk dot could gate the Siri action, for free

IntentAuthenticationPolicy = .requiresAuthentication · WWDC26 Session 347

Apple's own risk taxonomy for agentic features — Destructive/Exfiltration = High, Data Manipulation/Injection = Medium — maps directly onto the .dot.warn already on the Approval-waiting row. If Siri could ever act on that row, forcing Face ID for the high-risk case is the same decision the reinstated biometric gate already makes — just enforced at the OS level instead of only inside the app. One taxonomy, cited externally, instead of two invented separately.

W

A bigger Home Screen tile for the single top item — corrected framing

.systemExtraLargePortrait widget family · WWDC26 Session 277

Earlier research called this a "full-screen widget" — that framing doesn't hold up. The real capability is a larger portrait widget family (ported from visionOS), still placed like any Home Screen widget, not a Lock-Screen-covering surface. Still useful: it's a legitimate bigger presentation of the same Away Digest data Panel A already computes, reusing one data source at two sizes — not a new pipeline.
