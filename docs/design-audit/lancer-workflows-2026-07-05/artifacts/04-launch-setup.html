How someone actually uses it

From a link to a running mission

A

Share into Lancer, not into an inbox

A GitHub PR, Linear ticket, or Sentry link comes in through the iOS share sheet. Lancer attaches it and drafts a title straight into the composer — no separate evidence inbox to triage first.

B

Confirm the contract, not a plan

Tapping the composer expands a sheet of chips: repo, machine, agent, run mode, expected proof, interrupt rule. Three rows below let you open protected paths, done criteria, or interruption rules before tapping Start.

C

Set defaults once, reuse every launch

Mission Defaults holds run mode and limits (time, retries) as a standalone sheet reachable from the composer or a repo's playbook — edit once, every future launch chip picks it up.

D

The repo remembers its own rules

Working directory, proof command, PR branch, and protected zones live in a per-repo Playbook under Workspaces — operational config, explicitly not a notes app.

E

See what's missing before you commit

A preflight check states machine + agent readiness plainly, flags one blocking gap (no preview URL) without stopping you, and shows provider capability badges — resume, diffs, cost, image input — right where you decide to launch.

What Mobbin taught us, and why

Six references, one for each decision

Digg share/import stylePanel A

A shared link lands as an attached preview inside the destination surface, not a new intake screen. Confirms Panel A's decision to attach the PR link directly into the composer draft.

Formula 1 — "Confirm Chip" sheetPanel B

A locked-in choice shown as a single summary card with plain-language consequences, then Cancel/Continue at the bottom. Matches the launch contract's chip card plus its two-button decision bar exactly.

Lyft — "Set a check-in" radio listPanel C

Named behavior tiers (never / conditional / always) each with a one-line consequence, radio-selected. Directly informs Mission Defaults' Run Mode rows — "Normal" vs "Hands-off until proof" read the same way.

Mercury settings/checklist densityPanel D

Dense, labeled config rows (not cards, not prose) for values a person sets once and rarely revisits. This is why the Repo Playbook reads as a settings list — working directory, proof command, branch — instead of a document.

Grab Driver — Diagnostics checklist & N26 warning setupPanel E

Green checks for what's fine, one isolated warning row for what isn't, no modal blocking progress. Exactly the shape of the readiness check: two "online/available" lines, one amber preview-URL warning, and a "Continue" that still works.

Why this shape, not a bigger one

These stay inside the composer — never their own root

Every panel here is reachable from somewhere that already exists: the composer sheet, a Workspace's Playbook, or Settings. None of them is a destination in the sidebar. That's a deliberate, previously-made cut, not an oversight.

!
Explicit non-goal · carried from the wireframe's own notes

The board's own annotation for this section states it plainly: these views "should never become a Mission Draft root or agent-router dashboard." The underlying coding agent already plans its own work once launched — Codex, Claude Code, and OpenCode all do their own task breakdown. Lancer's job at launch time is the boundary and contract (repo, machine, proof, limits), not a second, competing plan of what the agent should do step by step.

A standalone "Mission Draft" screen would duplicate that planning surface and invite scope creep — a queue of drafts, draft templates, draft sharing — none of which this product needs. Keeping Launch Setup as depth under existing surfaces (composer → sheet → Mission Defaults / Playbook) keeps the sidebar at five destinations and the composer as the one place work starts.

iOS 27 / WWDC 2026 — what applies here

Two capabilities are a near-exact fit; one App Intent shape matters for launch itself

Cross-checked against the full platform research pass. Picked only what concretely changes a decision already made on this board — not every iOS 27 feature that could theoretically touch a composer.

FM

Draft the launch contract from a screenshot

Attachment(UIImage) + LanguageModelSession · WWDC26 Session 241/242

Foundation Models now takes an image directly in a prompt, on-device. A screenshot dropped into Panel A's share intake could draft the title, repo guess, and even a first-pass chip set for Panel B in one on-device call — no network round trip before the contract exists.

VS

Tap-to-segment for the attachment itself

GenerateIterativeSegmentationRequest · WWDC26 Session 237

The wireframe's own notes already call this out: Tap-to-Segment Bug Capture in Panel A reuses the same Vision mask interaction as the Work Thread's proof-annotation view. One capability, two use points — isolate the broken element at input time here, at output time there.

LR

A "Start Mission" Siri intent needs to outlive 30 seconds

LongRunningIntent + CancellableIntent · WWDC26 Session 345

If launching ever becomes a Siri/Shortcuts action, a mission plainly won't finish inside the standard 30-second App Intent window. LongRunningIntent hands progress to a system-rendered Live Activity with a built-in cancel button — worth designing for now if hands-free launch is ever scoped, since it changes what the intent has to report.
