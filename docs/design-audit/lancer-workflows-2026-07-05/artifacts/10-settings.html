How someone actually uses it

Setting a default once, not managing state daily

A

Open Settings — a grouped list, not a dashboard

An account line up top ("Local pairing only · no hosted billing"), then three quiet groups: General (appearance, notifications, security), Machines (pairing, diagnostics), Data (audit export, legal). No hero card, no stats.

B

Drill into Security & approvals

A one-line trust statement first ("Safety is always on"), then Approvals (biometric, default autonomy, policy audit log) and Trust (trusted machines, provider keys) as two small groups — not a governance dashboard.

C

Recover from denied notifications

A warm-toned banner names the problem and the fix ("Open iOS Settings"), then per-severity push defaults (Critical/High/Medium) below it — the same recovery pattern onboarding uses, reachable without repeating onboarding.

D

Switch Lancer accounts and vendor logins

Account Switcher hot-swaps between Lancer accounts (personal vs team) and per-vendor CLI identities without re-pairing machines. Active account sits at the top; vendor rows show which login each agent will use on the next run.

E

Compressed local audit summary

On-device audit digest uses Foundation Models to summarize audit.log on the phone — approval counts, drift flags, emergency stops — with follow-up questions that never leave the device. Distinct from raw log export or compliance bundles.

What Mobbin taught us, and why

Calm, trust-heavy, never a sales page

MLS settingsRoot structure

Calm grouped settings with account and preference rows. Confirmed against grouped-list settings screens (NYTimes Settings, MLS) — the pattern holds: account/plan first, then Feedback/Legal/Account-security groups, nothing competing for attention above the fold.

Revolut settingsTrust density

Dense trust/account settings that stay scannable despite covering a lot of ground. Directly informs the Security & approvals grouping (biometric, autonomy, audit log, trusted machines, keys) — adapt the density, not the finance-app upsell weight.

Luma settingsRestraint

A simple account/settings page with restrained chrome — the reference for why the Lancer root has no cards, no gradients, just rows.

Mercury security & Up securitySerious, plain-language

Security actions read as clear and serious without banking-compliance heaviness. A fresh search for biometric-toggle + trusted-device screens turned up the same shape independently — Afterpay's Face ID toggle + login-activity row and Stake's Passcode & biometrics + account-verification group both pair one toggle with one drill-in list, exactly the Panel B shape.

Why this shape, not the old one

Settings was doing seven jobs; it should do one

The current screen mixes policy/governance, provider keys, terminal-adjacent connection tools, relay pairing, diagnostics, plan copy, and legal into one scroll. The redesign's core move: push daily-work state (approvals queue, machine health, run status) back to Home, Workspaces, and Review, and leave Settings holding only defaults and account/trust controls.

iOS 27 / WWDC 2026 — what applies here

Two toggle-state branches Settings has to render correctly

Cross-checked against the full platform research pass, filtered to what's concretely a settings-row concern — not "AI could power a feature" speculation.

FM

On-device model availability is a real toggle state, not a boolean

SystemLanguageModel.availability · .appleIntelligenceNotEnabled

If Lancer ever surfaces an on-device Foundation Models option (e.g. local summarization of approval context), Apple requires branching on SystemLanguageModel.availability — not just checking device support. .appleIntelligenceNotEnabled is itself a user-facing iOS Settings state Lancer's own row has to detect and explain, the same way the notification-denied panel (C) already does for push permission.

NT

Notification grouping changed under the severity rows this screen configures

Same-app notification coalescing · no new interruption-level API

iOS 27 coalesces same-app notifications into one updating card instead of stacking them. No new formal API, but panel C's Critical/High/Medium push defaults are exactly what governs a burst of approval alerts — re-test how that burst actually renders under the new grouping before shipping, since the severities are configured here.

ET

A Settings-driven widget write needs the execution target set today

allowedExecutionTargets: ExecutionTargets · WWDC26 Session 345

Lancer already ships LancerWidget and LancerWatchWidget reading a shared WidgetSnapshot. If a future widget intent needs to change a Settings-owned default (autonomy preset, notification severity) rather than just read status, allowedExecutionTargets: .main is what routes that write into the real app process instead of a sandboxed widget-extension copy.

Full source doc: docs/design-audit/2026-07-05-ios27-wwdc26-platform-capabilities.md — none of these three block V1.
