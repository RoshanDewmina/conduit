How someone actually uses it

Decide without leaving the thread

A

The thread interrupts itself

An approval-waiting card appears inline in the Work Thread. Tapping Review diff drops in a sheet with request, scope (machine, repo, branch, files), and collapsed evidence — the matched policy rule, not raw command output. Deny sits right next to it; nothing here requires opening a separate app section.

B

Diff is a drill-in, not a tab

Only from the review sheet: file-by-file diffs with readable mobile line wrapping, red/green without horizontal scrolling. Approve patch stays contextual to what's on screen — there is no always-on code browser competing for space.

C

Deny has a middle option

Ask for changes sends a precise instruction back into the same thread instead of a silent reject — the agent resumes where it left off. Once decided, the surface flips to read-only: decision, actor, and timestamp persist, and the approve action is gone for good.

D

Policy changes get the same diff treatment

Policy Diff Review (reachable from Settings → Security) shows what changed between policy runs — red/green rule deltas, who edited, which machines are affected. High-risk governance edits require a second approver before they apply, using the same review anatomy as patch approval.

What Mobbin taught us, and why

Borrow approval hierarchy, borrow diff literacy, keep Cursor-dark chrome

Airwallex approval detailScope near decision

Keeps request scope and the approve/reject action close together on one surface, confirmed again in a fresh search. Directly informs why machine/repo/branch/files sit in the same review card as the decision bar, not a separate step.

Remote Global HR approvalActor & state clarity

Actor, request, and review-state read at a glance without extra taps. Basis for the resolved state in panel C showing decision, actor, and time as plain text rather than an icon-only status.

Revolut Business approvalProportional friction

Feels serious without visual panic — higher friction only for genuinely higher risk. Backs the proportional-gate rule: low risk stays quick, high/critical risk forces the diff or biometric before Approve enables.

GitHub mobile reviewDiff literacy

Changed-files summaries and inline diff are already familiar to developers. Panel B copies this shape (file header, +/− count, per-line color) while deliberately not importing PR comments or full review threads.

Manus diff/editor view Manus diff view · closest analog, not copied

Fresh search surfaced Manus's own patch view (red/green diff under a live coding-agent action) — the nearest existing product to "an agent showing you its own diff." It confirms the shape is sound but its chrome is generic-agent, not Cursor-dark; kept as validation, not a template.

Why this shape, not a heavier one

One anatomy, proportional friction, diff only when needed

The design bet: request, scope, evidence, and decision live on one reusable surface; the diff is a drill-in reached only from that surface, never a default view. That keeps low-risk approvals fast while still giving high-risk patches a real inspection step. The current-state audit found the code hasn't converged on that bet yet.

InboxBoardCard inline triage, InboxApprovalDetail sheet, and DSReviewSheet on Home are three separate presentations of the same decision today, with overlapping but not shared anatomy. This wireframe's whole premise — one reusable review surface — requires consolidating these into one component before it's true in code, not just on the board.

iOS 27 / WWDC 2026 — what applies here

Apple's own agentic-risk session is built for exactly this screen

Cross-checked against the full platform research pass. Approving a high-risk patch is the single best fit for this material anywhere in the app — it's literally the "Destructive"-tier action Apple's own taxonomy exists to gate.

AA

Approve as an authenticated intent, compared against the existing gate

IntentAuthenticationPolicy = .requiresAuthentication · WWDC26 Session 347

Apple's risk taxonomy classes destructive/exfiltration actions as High by default, forcing device authentication — the exact tier a high-risk patch approval sits in. Lancer already ships this instinct as a hand-rolled risk-tiered biometric gate (commit 695d2440, SecurityKit/BiometricGate.swift + ApprovalDecisionAuth.swift): the categories line up closely enough that Lancer's policy engine gains an externally-validated vocabulary to cite, not just an internally-invented one.

VA

"Hey Siri, explain this file"

.appEntityIdentifier(EntityIdentifier(for:identifier:)) · View Annotations API

Panel B's diff-file rows are exactly the "multiple meaningful items visible at once" case View Annotations targets. Tagging each file header lets a spoken reference to "this file" or "this diff" resolve to the specific changed file on screen, not a generic app launch. (The .appEntityIdentifier modifier shipped in iOS 18.4; what's new in WWDC26 is Apple's Siri integration guidance around View Annotations.)

DM

Deterministic-over-probabilistic validates the policy engine, not the prompt

.onToolCall / .historyTransform vs. "spotlighting" · WWDC26 Session 347

Apple explicitly downgrades prompt-level mitigation ("spotlighting" untrusted content) below deterministic gates that block execution outright. That's the same architecture choice behind panel A's "Matched rule" evidence — a daemon-side policy hold, not an instruction to the agent to be careful — giving Lancer's fail-closed design an external citation.
