How someone actually uses it

After proof passes, what you can actually do

A

Ship it

Proof passed unlocks three plain steps in order — Commit (message pre-drafted), Push, Create PR (proof summary attached automatically). No git vocabulary required to get from green tests to an open PR.

B

The merge gate

Merging a protected branch is never a single tap. CI-green and proof-attached are shown as already-true facts, then a distinct review card asks the actual question — method (squash), one extra confirmation, one audit entry — before Merge is live.

C

Flight recorder

Every mission collapses to a scrubbable timeline — contract, questions, proof chapters, commits — each with a timestamp and a one-line reason. Jump to event drills back into the thread at that exact point instead of re-reading the whole transcript.

D

Work search

One search field with status chips (Active/Merged/Failed/Proof) finds a mission by what happened to it, not just its title — a proof match, a diff match, or a mention buried in a digest all resolve to the same result row.

E

Share proof, not account settings

Contextual share/export for this mission only. Sync, billing, and audit export stay in Settings — no duplicate Account screen here. Share and export are separate, explicit, redacted-by-default actions — not folded into the sync banner.

F

Compliance bundle for security buyers

Compliance Export produces a signed, date-ranged audit archive — approvals, policy diffs, hash chain, machine fingerprints — distinct from panel E’s mission-scoped proof link. Reachable from Settings → Data and Ship → History context when exporting fleet records.

What Mobbin taught us, and why

GitHub's own merge gate, plus filtered search and honest billing

GitHub mobile merge sheetMerge gate

Review-required and checks-passed are shown as separate, already-resolved facts above the merge button itself — not folded into one vague "ready" badge. Directly informs splitting panel B's proof-status card from its protected-branch review card.

Forest timeline & Bevel timeline/history densityFlight recorder

A vertical line of timestamped events, each with a one-line detail, stays legible at a density of one event per meaningful action — the direct model for scrubbing through a mission instead of a raw log dump.

Google Drive filtered search & Squarespace search/historyWork search

Status/type chips sit directly under the search field and narrow results without leaving the screen — why Active/Merged/Failed/Proof are chips, not a separate filter sheet.

GoPro Quik account/billingSync & billing honesty

Real toggle state plus explicit billing rows (membership, amount, next date) — no invented aggregate number standing in for actual sync state. The pattern panel E's "Last synced 4m ago" line is built on.

Discord search resultsResult density

Mixed result types (messages, files, mentions) render as one scannable list with a one-line context snippet each — the shape behind panel D's three very different match kinds sharing one row style.

Why this shape, concisely

A continuation of Work Thread, and never a fake number

The board's own note is the argument: these five views "complete the loop after proof" and "should feel like continuations of Work Thread, not a separate release-management product." Ship actions live inside the same dark transcript chrome as the rest of the mission — same cards, same rail, same composer — because shipping is the last few turns of one conversation, not a handoff into a different app.

!
No fake metrics · the same rule that already has a documented bug elsewhere

The sidebar footer elsewhere in this app hardcodes Relay connected · 3 hosts regardless of reality — a documented, unfixed P0 trust bug. Sync and billing state in panel E are held to the same standard from the start: Last synced 4m ago is a real timestamp, Approval state remains host-owned is a true scoping statement (not "everything is synced"), and plan/share/export are separate rows, never one optimistic aggregate. Panel E is designed so it cannot regress into that bug's shape.

iOS 27 / WWDC 2026 — what applies here

Spotlight RAG for search, long-running merges, on-device drafting

SP

Work Search could go semantic, not just lexical

Spotlight-backed local RAG tool + IndexedEntity · Apple Intelligence guide

Foundation Models ships a built-in Spotlight RAG tool — "no embeddings, no vector database, no setup" — against the device's own Spotlight index. Marking mission/proof entities as IndexedEntity and querying that tool on-device is a zero-infrastructure path to "what did I ask about the checkout flow last month," not just a keyword match on panel D.

LR

Merge/ship can outlive the 30-second intent limit

LongRunningIntent + ProgressReportingIntent · WWDC26 Session 345

CI on a real repo can run well past 30 seconds. A merge or ship action modeled as a LongRunningIntent gets progress that automatically renders as a Live Activity with a system-provided cancel button — no hand-built polling UI needed to keep panel A/B honest while CI runs.

FM

The pre-drafted commit message can be genuinely on-device

SystemLanguageModel structured generation · WWDC26 Session 241/242

Panel A's "Suggested message ready" doesn't have to round-trip through the coding agent or a server call — the on-device Foundation Models session can draft a short commit message or PR summary from the diff directly on the phone, at no API cost and with nothing leaving the device.
