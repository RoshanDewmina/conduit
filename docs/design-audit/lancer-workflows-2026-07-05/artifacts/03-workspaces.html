How someone actually uses it

Choosing where work runs

A

Open Workspaces — a folder list, not a fleet

Top row is always All Repos with an aggregate needs-review count. Below it, one row per repo — name, current machine, and a quiet status line ("checks passed," "no active work"). Add Repo sits at the bottom as the last row, not a floating button.

B

Tap into a repo

Same ledger shape as Home, scoped to one repo: Today's runs and approvals, then Run targets — the machines that can actually execute work here, each showing which agents are installed.

C

Check on a machine

Tap a run target and a sheet drops in: online status, which agents are available, trust/fingerprint, then Diagnostics and Remove Machine below a visual break — separating "look" from "act."

D

Something's unreachable

An offline machine doesn't hide in a detail sheet — it's promoted straight into Needs you with a "Fix" action, tells you exactly what still works (cached threads) versus what's blocked (new runs), and names a likely cause before you go looking.

E

Add a repo or a machine

One Add sheet, three rows: a path on a machine you already trust, a fresh code-pairing flow (same mechanism as onboarding), or a cloud runner if hosted agents are on. One setup action family instead of two separate flows.

F

Spot policy drift before it bites

Cross-host policy-consistency check surfaces when two machines running the same repo disagree on governance rules — a warm banner on the workspace detail, per-machine drift badges, and a one-tap path into Policy Diff Review. Fleet trust shouldn’t silently diverge.

What Mobbin taught us, and why

Two different jobs, two different reference sets

Slack workspace switcher & monday.com workspace listList shape

Simple identity rows that stay legible even as the object count grows. Why the top-level list is folder rows with counts and a chevron — nothing heavier.

GitHub repository listRow metadata

Repos are scannable with owner/state metadata, not a dashboard tile per repo. Directly informs the "checks passed" / "no active work" one-line meta under each name.

Evernote connected devices & Telegram connected devicesTrust & revoke

Device identity and management actions (remove/revoke) shown clearly, but only once you're already looking at one device — never on the list itself. This is exactly why remove/diagnostics live in the machine detail sheet, not the workspace row.

Starlink Network → DevicesDeveloper-trusted framing

Dense, technical host rows built for people who already understand what a "host" is — a stronger model for machine rows than consumer smart-home device lists. The explicit warning attached to this reference: don't turn Lancer into an SSH client just because the audience overlaps.

Why this shape, not the old one

The old Machines screen had three contradictions, not just too much chrome

Machines used to be a fleet dashboard: every paired host as a card, vendor spend, health metrics, up front. Beyond being noisier than Cursor's Workspaces reference, the current-state audit found real correctness problems this redesign is built to close, not just restyle.

!
Contradictory connection copy · P0 trust, documented

The sidebar footer hardcodes Relay connected · 3 hosts regardless of reality, while the Machines screen shows whatever host is actually seeded — two different claims about the same connection state, visible on screen at the same time. A workspace-first redesign doesn't fix this by itself; it still needs the sidebar footer wired to real counts.

Two more, lower-severity but real: the relay path (one live host) and SSH slots (FleetStore) currently use different status semantics for what should be one machine-status vocabulary, and onOpenTerminal still exists as a drill-in from the fleet view — a live terminal entry point that must not reach V1 navigation. Neither is solved by the Workspaces relabeling alone; both are implementation-gate items, not design decisions.

Open decision — not yet resolved

"Workspaces" is more than renaming a screen

The nav-discrepancy investigation (from the Command Home artifact) flagged this directly, and it lands squarely on this workflow: the current Machines view is genuinely host-first (FleetView lists machines, repos are inside them); the wireframe wants repo-first (workspaces list repos, machines are a property of each one). That's the reverse of the current data model, not a label swap on the same screen.

Real decision
Repo-first vs. host-first is a data-model change Current: FleetView.swift + FleetStore.swift, organized by machine. Proposed: organized by repo, with machines as a property of each workspace. This needs an explicit product decision before implementation, not just a visual pass.
Doc still says "the code route can remain .machines"
The workflow doc already anticipated a staged rename Its own recommendation: rename the visible product surface now, migrate the code route name later. That's a reasonable sequencing call, but it should be made explicitly, not discovered mid-implementation.
Corrected in this artifact
Pairing entry point file was stale The workflow doc cites BridgePairingView.swift, which no longer exists. The real current file is Packages/LancerKit/Sources/OnboardingFeature/OnboardingPairing.swift — verified directly against the repo before writing this page, not copied from the doc.

iOS 27 / WWDC 2026 — what applies here

Trust actions are exactly what Apple's own risk model is for

Cross-checked against the full platform research pass. Workspaces has a sharper fit than Home for one of these — pairing and removing a machine are literally the "Destructive"/trust-elevating actions Apple's own agentic-security session names.

AA

Pair / Remove Machine as authenticated intents

IntentAuthenticationPolicy = .requiresAuthentication · WWDC26 Session 347

Apple's own risk taxonomy calls a destructive, trust-changing action "High" by default. Remove Machine is exactly that, and Pair Machine is its inverse (granting trust, not just removing it) — both are stronger candidates for this policy than almost anything on Home, since they permanently change what a machine can do, not just what's on screen right now.

VA

"Hey Siri, is hermes-box online?"

.appEntityIdentifier(EntityIdentifier(for:identifier:)) · View Annotations API

Workspace and machine rows are, again, the "multiple meaningful items visible at once" case. Tagging each machine row lets a status query resolve to a specific host instead of a generic app launch — genuinely more useful here than on Home, since "is X online" is a question people actually ask about infrastructure.

ET

A future machine-status widget needs this from day one

allowedExecutionTargets: ExecutionTargets · WWDC26 Session 345

If a Home Screen widget ever shows machine status with a "reconnect" button, that write has to land in the main app's data model, not a sandboxed widget-extension copy — this API is the explicit fix for exactly that failure mode. Worth deciding at design time, not discovering after a widget ships silently broken.
