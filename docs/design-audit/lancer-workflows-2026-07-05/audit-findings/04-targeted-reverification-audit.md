# Targeted Re-Verification Audit — Findings

Scope: 9 workflow artifacts (skipping Onboarding, covered separately). Pass 1 = iOS 27/WWDC26 claims vs. inline WWDC 2026 platform notes in each artifact (source research purged 2026-07-06) + live Apple docs. Pass 2 = Mobbin citation strength, with fresh `mcp__mobbin__search_screens` spot-checks.

## Pass 1 — Apple/iOS 27 claims

**artifacts/09-platform-gaps.html — real issue.** The Platform Intelligence panel (Panel C, "One widget template, four states") twice uses the phrase "full-screen widgets" ("Three named full-screen widgets plus fleet-wide status collapse into one template," and again in the prose block: "3 separate full-screen-widget ideas") but never once anchors this to the source doc's own correction — `.systemExtraLargePortrait`. Every other artifact that touches widgets (artifacts/02-home.html, artifacts/07-fast-follows.html, artifacts/10-settings.html) explicitly performs the correction ("Earlier research called this a 'full-screen widget' — that framing doesn't hold up..."). This is the one flagship widget panel across the whole set and it's the single place the corrected API name is missing. Fix: add the same correction sentence + `.systemExtraLargePortrait` citation to this panel, since it's the highest-risk spot for an engineer to build against the wrong mental model of a Lock-Screen-covering surface.

**artifacts/02-home.html, artifacts/03-workspaces.html, artifacts/06-review-diff.html — minor/verified-but-mis-framed.** All three cite `.appEntityIdentifier(EntityIdentifier(for:identifier:))` / "View Annotations API" as something new for iOS 27 (matching the source doc's own §1 framing, "View Annotations API (new)"). Live-fetched Apple docs show the modifier's actual platform floor is **iOS 18.4+ / macOS 15.4+** — a full cycle before iOS 27, not new in this release. The code shape itself is real and matches Apple's own sample exactly (verified via `mcp__apple-docs__get_apple_doc_content` on `swiftui/view/appentityidentifier(_:)`), so nothing is hallucinated — but "new," "iOS 27 is still in beta so no one's shipped this" (home.html) reads more confident/novel than warranted. What's actually new for WWDC26 is the *named pattern and Siri-integration guidance* around the modifier, not the modifier itself. This is a source-doc-level framing issue that propagated identically into all three artifacts; worth a one-line correction in each ("this modifier itself has shipped since iOS 18.4; what's new is Apple's View-Annotations guidance for using it with Siri").

**Verified clean via direct API lookup (no issues found):**
- `IntentAuthenticationPolicy` / `.requiresAuthentication` — real enum, confirmed cases (`requiresAuthentication`, `requiresLocalDeviceAuthentication`); source doc correctly does *not* claim this enum itself is new (only the Session 347 emphasis is new) — all citing artifacts (home, workspaces, review-diff, settings) match this correctly.
- `GenerateIterativeSegmentationRequest` — real, Beta, iOS 27+, confirmed in Vision framework (thread.html, launch-setup.html citations check out).
- `LongRunningIntent` + `performBackgroundTask` — real protocol/method, confirmed (launch-setup.html, ship-history.html).
- `allowedExecutionTargets: ExecutionTargets` — initially looked like a naming error (canonical type is `IntentExecutionTargets`), but confirmed `AppIntent.ExecutionTargets` is a real nested typealias for it (iOS 27 Beta) — the citation as written (workspaces.html, settings.html) is valid shorthand, not wrong.
- `isDynamicIslandLimitedInWidth`, `LanguageModelSession.DynamicProfile`, `SystemLanguageModel` platform table (iOS/iPadOS/macOS/Mac Catalyst/visionOS 26.0+, watchOS 27.0+ Beta) — all confirmed to match source doc exactly (fast-follows.html, ship-history.html, settings.html).

**Supplementary note, not an artifact-level fix:** the source doc claims `GenerateIterativeSegmentationRequest` platform support includes "watchOS (new in this cycle)," but Apple's live platform table for both the request class and its `RequestDescriptor` case lists only iOS/iPadOS/Mac Catalyst/macOS/tvOS/visionOS — no watchOS. No artifact repeats the watchOS claim directly, so this doesn't need a per-artifact fix, but the source doc's own claim is unconfirmed by the primary source.

**Artifacts that check out clean on Pass 1:** artifacts/04-launch-setup.html, artifacts/05-work-thread.html, artifacts/07-fast-follows.html, artifacts/08-ship-history.html, artifacts/10-settings.html.

## Pass 2 — Mobbin citations

**artifacts/03-workspaces.html — weak/unverifiable citation.** "Tailscale devices & Termius host list" (lesson: "Developer-trusted framing") is the only cite-card in the entire 9-artifact set with no `mobbin.com` URL at all — just plain text sitting inside a "What Mobbin taught us" grid otherwise made entirely of linked citations. A fresh `mcp__mobbin__search_screens` pass for "dense technical device/host list for developers, SSH-style" returned Starlink, Apple Home, Google Home, and Telegram screens — no Tailscale or Termius results surfaced at all. This citation looks like it was written from general product knowledge rather than an actual Mobbin lookup. Suggested fix: either replace with a verified link (Starlink's "Network → Devices" screen, `https://mobbin.com/screens/e7b43201-3ade-493f-8672-76efbd8bff8f`, is a reasonable real analog for dense developer-facing host rows) or drop the claim.

**artifacts/06-review-diff.html — real citation, just missing its link.** "Manus diff/editor view" (lesson: "Closest analog, not copied") has no URL, despite its own text claiming "Fresh search surfaced Manus's own patch view." A fresh search for "code diff view... inside an AI agent task screen" independently surfaced real, linkable Manus screens matching the description exactly (App.tsx diff with Diff/Original/Modified toggle) — e.g. `https://mobbin.com/screens/6834fa59-ee3a-4969-a6cc-c27bb0c9ad56`. Not a fabrication, just an easy completeness fix: add the link.

**artifacts/10-settings.html — partially unverifiable sub-claim.** "MLS settings" cite-card text adds "Confirmed against a fresh Mobbin pass of grouped-list settings screens (Zocdoc, Finimize, NYTimes)" — only MLS has an actual link; Zocdoc, Finimize, and NYTimes are named with no links. A fresh re-search for grouped iOS settings screens did surface a real NYTimes settings screen (`https://mobbin.com/screens/f97269c5-1cd0-46c1-abaf-0260e0699847`, confirming that one), but did not surface Zocdoc or Finimize in the results. Not proof they're wrong, but the two remain unconfirmed and should either get their own links or be dropped from the claim.

**Artifacts that check out clean on Pass 2 (all cite-cards linked, reasonably specific to the design decision cited):** artifacts/02-home.html, artifacts/04-launch-setup.html, artifacts/05-work-thread.html, artifacts/07-fast-follows.html, artifacts/08-ship-history.html, artifacts/09-platform-gaps.html.

## Summary

Real issues found: 1 in Pass 1 (platform-gaps.html's widget panel never applies the "full-screen widget" correction that every sibling artifact applies), 1 soft mis-framing repeated across 3 artifacts (View Annotations/`appEntityIdentifier` presented as new when the modifier itself predates iOS 27 by a cycle), and 3 weak/unlinked Mobbin citations (1 likely not actually sourced from Mobbin, 1 real-but-unlinked, 1 partially unconfirmed). No hallucinated APIs were found — every specific API name checked (`IntentAuthenticationPolicy`, `GenerateIterativeSegmentationRequest`, `.appEntityIdentifier`, `ExecutionTargets`, `LongRunningIntent`, `isDynamicIslandLimitedInWidth`, `DynamicProfile`) is real and behaves as described.
