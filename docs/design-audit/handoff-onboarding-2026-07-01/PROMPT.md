# Design brief: Lancer onboarding redesign (workflow 1 of 6)

You're redesigning one screen of a native iOS app. Several screenshots and
source documents are attached — look at the screenshots before anything
else, they're the ground truth. Read this whole brief, then **ask me
clarifying questions before you start designing.**

## What's attached

**4 screenshots** of the current app:
- One shows a terracotta gradient hero card, three icon+text rows below it,
  and a 6-digit code entry field — that's the screen you're redesigning.
- The other three show what comes right after it: an account-choice screen,
  an offline-name-entry screen, and the screen a user lands on right after
  onboarding finishes. You're not redesigning these three — they're there so
  you don't propose something that clashes with what follows.

**6 source documents** — the full audit work this brief is distilled from
(an audit packet, the detailed onboarding findings, a competitive-pattern
research log, two session write-ups, and a final summary). This brief has
what you need to start; go to these only if you want to verify something or
dig deeper.

## What Lancer is

Lancer is mobile mission control for AI coding agents (Claude Code, Codex,
OpenCode, Kimi) running on the user's own machine via a resident daemon. The
phone steers and approves; it is not a phone IDE. The product framing is
"warm control room" — a quiet, high-trust command console, not a chat clone,
not a developer toy, not a crypto/security dashboard.

## What you're producing

An **interactive wireframe** — real HTML/CSS or React, clickable/scrollable,
not a static image. It needs to be translatable into SwiftUI, so favor real
component structure (cards, lists, buttons with real states) over decorative
flourishes that don't map to anything. A single self-contained HTML file is
fine if that's easiest to share back.

## The roadmap (context only — you're doing step 1 today)

Two independent design audits reviewed six workflows in this app. We're
implementing fixes one workflow at a time: **1) Onboarding (today) → 2) Work
Thread → 3) Review/Approvals → 4) Machines → 5) Home & Settings.** You don't
need detail on the other five — just know this is one part of a larger pass.

## The screen you're redesigning

**Onboarding step 1: "Value + Pair"** — the first thing a new user sees.
Today it's: a hero card (app icon, italic kicker, bold title, one line of
body copy) → three abstract value-proposition rows ("Approve actions from
afar," "Watch the terminal stream live," "Policy guardrails per machine") →
a "run `lancerd pair` on your desktop" instruction with a 6-digit code field
(the code is generated by that command on the user's computer, then typed
in here — no QR/camera path, that's a deliberate product decision, not
something to reconsider) → a fixed "Pair & continue" button.

## The actual problem (both audits agree — this is the real brief)

1. **The user never sees the product before being asked to trust it with
   machine access.** The three value rows are abstract marketing copy, not
   proof. Nothing on this screen shows what Lancer actually looks like in
   use.
2. **Pairing failures are easy to miss.** A wrong/expired code shows as a
   small, muted, centered status line, not visually connected to the code
   field the user is looking at.

That's the brief. How you solve it is genuinely open — propose whatever you
think actually solves those two problems. If it helps to know what's already
been considered: a real product-preview visual instead of the abstract rows,
and moving the error message to sit right next to the code field, are two
ideas already on the table — not decided, not the only options, just so you
know that ground has been covered if you want to go further or somewhere
else entirely.

## Out of scope

- Anything after step 1 (account choice, policy step, SSH setup).
- The code-only pairing mechanic itself — that mechanism is fixed, only its
  presentation is in play.

Whatever you need — but I'd specifically value your take on how literal a
