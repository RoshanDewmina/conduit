3 Approaches — How Lancer Connects to Agents

All three support the same end-goal: any agent → Lancer cloud → your phone. The difference is the developer experience and what you're betting on.

APPROACH A

SDK Platform

REST + WebSocket backbone, thin SDKs per language

# Any agent, any language
lancer.notify("trade_executed", {
  ticker: "AAPL",
  action: "buy",
  amount: 500
})

# Block until user responds
decision = lancer.checkpoint({
  message: "Sell TSLA?",
  context: news_summary
})
✓ Universal — Python, Go, TypeScript, anything
✓ Bidirectional — interventions flow back in real-time
✓ Familiar DX — devs know REST + SDK patterns
✗ Need SDKs for each language
✗ Extra plumbing for LLM-native agents
Think: PagerDuty + Pushover + intervention layer + agent marketplace
APPROACH B — RECOMMENDED

MCP-Native First

Lancer exposes a cloud MCP server. Agents add it to their --mcp-server config.

# claude agent config
--mcp-server lancer-cloud

# LLM calls naturally:
lancer_notify(event="trade",
  ticker="AAPL", action="buy")

# Blocks until user taps phone:
lancer_checkpoint(
  message="Sell TSLA?",
  context=news_summary
)
✓ Zero friction for Claude/Codex users (already use MCP)
✓ LLM calls tools naturally — no extra SDK import
✓ Bidirectional — checkpoint() blocks until you tap
✓ Fits the AI agent ecosystem direction
✗ Non-LLM agents need a thin HTTP shim
✗ MCP protocol still evolving
Think: "The MCP server that connects every agent to your phone"
APPROACH C

Hybrid (MCP + SDK)

Both paths. MCP for AI agents. REST/WebSocket SDK for everything else. Same backend.

# AI agent → MCP path
--mcp-server lancer-cloud

# Python script → SDK path
from lancer import Agent
agent = Agent(api_key="...")
agent.notify("portfolio_alert", {...})
decision = agent.checkpoint(...)
✓ Best coverage — nothing excluded
✓ Natural fit per agent type
✓ Future-proof
✗ Double the surface area to build + maintain
✗ Risky to ship both at once
Verdict: Correct end-state, but build B first, then add SDK as a second path. Ship B, not C.

Why MCP-native first makes sense right now

Your primary users
Developers already running Claude Code / Codex agents. They are MCP users by default. Lancer becomes one more MCP server in their config — no new concept to learn.
The IBKR agent
claude --mcp ibkr-mcp --mcp lancer-cloud
The agent can now call lancer_notify() naturally. No Python SDK, no extra process.
Lancer's moat
Being "the MCP server for mobile" is a clear, defensible position in the ecosystem. As MCP adoption grows, Lancer grows with it. The SDK path can always be added later.

Click to select an approach. My recommendation is B — start MCP-native, add SDK path once the platform is stable.
