the hypothesis

Four capabilities bundled into one product

01

Enforced verification

Refuse an agent's "done" / "tests pass" claim unless a real, independently-run build or test artifact backs it up.

false-green refusal, not self-reported success
02

Sim broker / leases

A cross-process broker hands out exclusive simulator ownership so concurrent agents never grab the same device.

queue + lease, not "hope nobody collides"
03

Host-state isolation

Each agent's CoreSimulator state, DerivedData, and module cache are walled off from every other agent on the box.

per-agent sandbox, not one shared Mac
04

Device Hub orchestration

Allocates and schedules many simulators/devices across many agents through Xcode 27's new Device Hub + devicectl.

fleet policy, not manual boot/shutdown

closest contender 1 / 3
SandVault
webcoyote — OSS · 348★ · 17 forks · 420 commits, actively maintained

Runs Claude Code / Codex / Cursor Agent / Gemini in a limited macOS user account + sandbox-exec. Boots a fresh scratch simulator per session, deleted on exit.

Covers

  • Host-state isolation — separate user account, no access to your home dir or other volumes.
  • Per-session sim isolation — a fresh sandvault-<session-id> simulator is created and destroyed for each run.
  • Works today, zero VM overhead, free.

Gaps

  • No verification gate — explicitly built for autonomy (--dangerously-skip-permissions, --yolo), not proof-of-work.
  • No cross-session broker — one sim per session, not a shared pool with queueing/contention policy across many concurrent sessions.
  • No Device Hub fleet view — and GUI apps (incl. Simulator.app) can't run inside the sandbox at all, by its own documentation.

closest contender 2 / 3
XcodeBuildMCP
Cameron Cooke, acquired by Sentry (2026) — the de facto standard agent↔Xcode bridge

82 MCP tools spanning build, test, simulator control, and LLDB debugging. The tool most agents already reach for to "make Claude ship iOS code."

Covers

  • Breadth of control — build/test/screenshot/debug/UI-automation in one contract, no raw xcodebuild wrangling.
  • Backed by a company (Sentry) — real maintenance, not a weekend project.
  • The widest adoption of any tool surveyed.

Gaps

  • No verification gate — reports whatever the agent asked it to run; can't stop an agent from claiming success it never checked.
  • No leasing — session_set_defaults implies one active simulator/session context per client, not multi-tenant concurrency control.
  • No isolation — runs straight against the host's shared CoreSimulator/DerivedData; two agents can collide freely.

08 / 13
closest contender 3 / 3 · plus one to watch
Apple xcrun mcpbridge
Xcode 26.3+ — first-party MCP server, ships inside Xcode itself

Zero-config: auto-detects the running Xcode PID and exposes 20 tools (build, test, Swift REPL, SwiftUI preview, simulator control) over XPC.

Covers

  • Native Device Hub access — the only tool here with first-party, zero-latency access to Apple's own simulator/device model.
  • No install, no drift from Xcode's own build graph.
  • Honorable mention — Baguette (OSS, 16★): headless multi-sim boot/stream/inject farm — the closest thing to a fleet-orchestration primitive, but zero isolation or leasing logic.

Gaps

  • Explicitly single-instance — built for "the typical single-instance case," by Apple's own framing. No concurrency model.
  • No isolation, no verification — shares the one Xcode/CoreSimulator instance; reports what Xcode reports, nothing independently checked.
  • Fastest-moving target in this whole audit — Apple ships this monthly.

recommendation

No-Go on the bundle. Go on a wedge.

NO-GO

A funded, standalone company selling all four pillars bundled as one iOS-only product.

  • Apple is actively closing the orchestration + native-bridge gap itself.
  • Verification is better solved horizontally — that's where Qodo/Baz money already went.
  • Buyer intersection (iOS × agents × parallel × Mac × already in pain) is small today.
GO — small

A narrow, open-source sim-broker/lease layer that plugs into XcodeBuildMCP / Apple's mcpbridge rather than replacing them.

  • Take SandVault's per-session scratch-sim trick and generalize it into a real cross-process broker for N concurrent agents, not just N sandboxes.
  • Ship free, build a user base the way SandVault/Baguette already have.
  • Defer verification-gate (solve generically, don't rebuild Qodo for iOS) and Device Hub fleet UI (let Apple/Bitrise own it) until adoption proves out.

top risks

Six reasons to stay small

RISK 01

Platform-owner risk

Device Hub + xcrun mcpbridge both shipped in 2026 alone. Multi-instance support is a plausible next Apple feature, not a moat.

RISK 02

TAM ceiling

iOS-only, Mac-only, "multiple concurrent agents on one repo" — a small population even inside the agentic-coding boom.

RISK 03

Free substitutes exist

SandVault (348★), Baguette, ClodPod, Chamber are live, OSS, and actively developed — a paid product competes with "good enough and free."

RISK 04

Verification is a horizontal win

Qodo ($70M) and Baz ($17M) prove the money is in general code verification, not an iOS-scoped slice of it.

RISK 05

Private-API fragility

CoreSimulator/SimulatorKit internals break every OS point release — Baguette's README documents this happening between iOS 26 builds.

RISK 06

Local-only distribution caps upside

Cloud CI vendors (Bitrise, Codemagic) can add Device Hub-aware orchestration server-side and out-execute a local Mac tool for team scale.

corroborating signal

One real team already hit every symptom

Sim HID went dead mid-session — "iOS 27 Simulator HID/accessibility is fully non-functional," confirmed via a control-test tap that produced zero response.
A real race only surfaced under live sim testing — a connection race in production code was found only by dogfooding on Simulator, not by unit tests.
Per-run git-worktree isolation, already built — agent.worktree.create/remove gives each dispatched agent its own path under ~/.lancer/worktrees/, distinct from vendor scratch dirs.
A "false-green" verification gate already exists in the workflow — the project's own agent contract requires evidence (build output, test counts) before any task is marked done.
honest caveat

This is one power-user team's own tooling, not independent market proof. It confirms the pain is real — it does not confirm anyone would pay for a bundled product to fix it.
