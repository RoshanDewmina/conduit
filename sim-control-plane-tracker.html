02 · review

Critique of the report (agreed in discussion)

Stack rec was backwards

Report recommended Swift "to share Lancer's stack" — but Lancer's daemon is already Go. Premise false.

Latency is the real killer

Fresh DerivedData per run ⇒ 5–15 min verifies. Kill criterion #3 (users disable the gate) is the likeliest death. Needs tiered policies from day one.

Freshness undefined

"Any relevant source change invalidates evidence" — relevant was never designed. Naive hashing self-invalidates on README edits or proof writes.

Leases are advisory

A UDID lock only binds cooperating processes. Raw simctl / Xcode ignore it. Matrix overstated "machine-wide" enforcement.

Also: §13.1 blurred hooks vs. hard gates — hooks block an agent's stop, only your own accept command hard-blocks.

04 · product definition (current)

A local control plane for sims & devices
shared by concurrent coding agents

L1

Sim broker / leases v0 core

Resident daemon owns exclusive leases: acquire / queue / renew / release, PID + heartbeat, stale reclamation, capacity limits. Queue + lease, not "hope nobody collides". The report's truest whitespace — nobody ships this.

L2

Host-state isolation v0 thin

Lease grant returns an isolated env: per-lease DerivedData, module cache, result bundles, optional cloned sim. ⚠ VibeChard's turf — do minimal redirection, differentiate on the broker, or integrate VibeChard as backend.

L3

Device Hub orchestration v1

Scheduling policy on top of leases: pool sizing, queue fairness, RAM/thermal throttling, devicectl for physical devices. Fleet policy, not manual boot/shutdown.

L4

Proof gate deferred

Optional layer later, built on leases — cleaner dependency order than the report's v0.

05 · headline use case

N agents, N clones of the same iPhone, one Mac

The scenario

  • Agents A / B / C each build a feature in their own git worktree
  • Each acquires a lease → broker boots (or reuses from warm pool) a separate clone of "iPhone 17 Pro" via simctl clone — own UDID, own data container
  • Each builds with per-lease DerivedData, installs & UI-tests on its sim — no cross-contamination possible through the broker
  • Release / crash → clone cleaned or returned to pool

Limits & non-goals

  • Bottleneck is hardware, not CoreSimulator: ~1–3 GB RAM per booted sim; parallel xcodebuilds saturate CPU first
  • Realistic: 2–4 truly parallel build+test agents per Apple Silicon Mac → broker enforces capacity, agent #5 queues
  • Does not parallelize one feature's test suite (one sim per agent)
  • Source isolation stays on worktrees; broker isolates the Apple-side state worktrees can't

06 · interface sketch

How an agent uses it

# agent (via MCP tool or CLI)
$ simplane lease acquire \
   --device "iPhone 17 Pro" --runtime 27.0

lease-86dd granted
  udid: 4E2A…9F
  env:  DERIVED_DATA=…/leases/86dd/dd
  ttl:  30m (heartbeat to renew)

# build/test inside the lease env…

$ simplane lease release lease-86dd

Agent-first adoption (Claude Code, Cursor, Codex)

  • MCP server (primary): lease_acquire / lease_release / lease_status — result carries UDID + env paths into agent context
  • CLI + JSON: for bash-driven agents, hooks, CI
  • PATH shim (opt-in safety net): intercepts xcodebuild/simctl so an agent that forgets to lease gets auto-scoped, not collided
  • Ship a paste-ready CLAUDE.md / rules snippet — agents only use tools they're told about; this is product, not docs garnish
  • Daemon: Unix socket, versioned JSON; same-UDID acquire queues; dead PID / missed heartbeat → reclaim + clean

07 · competitive position

Who's nearby, and the gap we occupy

VibeChard

Best-in-class per-task isolation (DerivedData, caches, sim clones). No machine-wide broker, no queue, no cross-tool leases. Integrate, don't rebuild.

XcodeBuildMCP

Agent build/test/run execution layer. Per-workspace daemon, no global resource authority. Potential execution backend.

Apple Device Hub + devicectl

Inventory, control, diagnostics, scriptable JSON. No task ownership, leases, fairness, or policy. Sit above it, don't clone it.

The gap: a machine-wide authority that owns which agent may touch which simulator, when. Per the report's matrix, no shipped tool has cross-process leases.

08 · open questions

To resolve next

1 · Lease adoption

PATH shim vs. explicit CLI/MCP vs. both. Lean: both, shim opt-in (shim is fragile across Xcode versions).

2 · Isolation depth v0

Env-var redirection only, or sim cloning too? Redirection covers the worst collisions (build.db races); clones cost disk + boot time.

3 · VibeChard relationship

Integrate as isolation backend, coexist politely (detect its env), or reimplement the minimal subset?

4 · Pool strategy

Warm pool of pre-booted clones (fast acquire, idle RAM) vs. boot-on-demand (~30–60s first acquire). Lean: on-demand first.

5 · Remote / hosted sims

Make the lease protocol location-transparent (UDID + connection info, host-agnostic). Ladder: local → your own second Mac → hosted fleet only with demand. Hosted competes with Limrun; Apple EULA caps 2 macOS VMs per Mac — thin margins, real capex. Not v0.

6 · Validation demo

AI-heavy multi-agent iOS devs. The demo that sells a broker: two agents collide live → broker prevents it.
